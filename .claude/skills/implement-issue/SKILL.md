# Implement Issue

## 概要

GitHub Issueの内容をもとに、BDDスタイルのテストケース設計 → ユーザーレビュー → TDD実装 → simplifyによるコード整理、という一貫したフローで実装を行う。

## ワークフロー

### Step 1: Issueの取得

引数がissue番号の場合とURLの場合を判定して取得する:

```bash
# issue番号の場合
gh issue view <番号> --json title,body,labels

# URLの場合もghが自動判別するので同じコマンドで動作する
gh issue view <URL> --json title,body,labels
```

取得したissueのタイトル・本文・Acceptance Criteriaを把握する。

### Step 2: テストケースの設計（実装前）

**重要: この段階ではテストコードの実装も、ソースコードの実装も行わない。テストケースの構造のみを出力する。**

Issueの内容（特にAcceptance CriteriaとBackground）を読み解き、以下のルールでテストケースを設計する:

#### テストケースの記述ルール

- `describe` と `test.todo` のみを使用（実装は書かない）
- **BDDスタイル**: 「〜のとき、〜であること」という日本語で振る舞いを記述
- 各 `test.todo` にコメントで **テストの意図** を記述（なぜこのケースが必要か）
- エッジケース・境界値・制約条件を必ず含める
- **モックを前提としたテストは書かない**（実際の依存を使う）

#### テストケースの構造例

```typescript
describe("予算APIの正常系", () => {
  test.todo("有効なグループIDと年月を渡したとき、その月の予算が返されること")
  // 意図: 基本的なGET操作の確認。正常なリクエストが通ることを保証する

  test.todo("予算が未設定のとき、404が返されること")
  // 意図: リソース未存在時のエラーレスポンス仕様の確認
})

describe("予算APIのバリデーション", () => {
  test.todo("yearに不正な値(文字列)を渡したとき、400エラーが返されること")
  // 意図: 型バリデーションが機能していることの確認

  test.todo("monthに範囲外の値(0や13)を渡したとき、400エラーが返されること")
  // 意図: 境界値チェック。月は1〜12の範囲制約があるため
})
```

#### テストケース設計のチェックリスト

設計後に自己チェックする:
- [ ] Acceptance Criteriaのすべての項目がテストケースに対応しているか
- [ ] 正常系・異常系・境界値がカバーされているか
- [ ] 「〜のとき、〜であること」形式になっているか
- [ ] モックに依存したケースが混入していないか
- [ ] 各テストに意図コメントがあるか

### Step 3: テストケースのユーザーレビュー

設計したテストケースをユーザーに提示し、AskUserQuestion で確認を取る:

```
以下のテストケースを設計しました。実装に進んでよいですか？
修正が必要な場合は教えてください。

[テストケース全文]
```

- **OKなら Step 4 へ進む**
- **NGなら** 指摘を受けてテストケースを修正し、再度確認を求める

### Step 4: TDDによる実装

`tdd-workflow` スキルを使って赤-緑-リファクタリングのサイクルで実装する:

1. **赤フェーズ** (`tdd:red`): `test.todo` を実際のテストコードに変換して失敗させる
2. **緑フェーズ** (`tdd:green`): テストが通る最小限の実装を書く
3. **リファクタリングフェーズ** (`tdd:refactor`): テストを保ちながらコードを整理する

テストファイルの配置は `test/` ディレクトリ、ファイル名は `*.test.ts`。

### Step 5: コードのSimplify

実装完了後、`simplify` スキルを使用してソースコードを整理する。

## 注意事項

- テストケース設計とユーザーレビューは必ず Step 4 の前に完了すること
- モック利用を前提としたテストはこのプロジェクトのポリシーに反するため書かない
- Acceptance Criteriaがissueにない場合は、BackgroundとOverviewから推測してテストケースを設計する
