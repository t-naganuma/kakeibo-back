# Create Issue

## 概要

引数で受け取ったissue内容をもとに、ユーザーへの複数回のインタビューを通じて詳細を深堀りし、プロジェクトのISSUE_TEMPLATEフォーマットでGitHub Issueを作成する。

## ワークフロー

### Step 1: 引数の評価

引数（`<issue内容>`）を受け取ったら以下を評価する:

- **内容が具体的か** — 実装対象・バグの再現手順・目的が明確か
- **受け入れ基準が導けるか** — 完了条件を定義できるか
- **リスクや依存関係が見えるか** — 他のissueや機能との関係が分かるか

いずれかが曖昧な場合は **Step 2のインタビュー** に進む。十分明確な場合は Step 3 に進む。

### Step 2: ユーザーインタビュー（複数ラウンド）

**重要: インタビューは必ず複数回行うこと。** 1回で終わらせず、段階的に深堀りしてユーザー自身に気付きを与える質問をする。

#### ラウンド1: 全体像の把握

AskUserQuestion ツールで以下のような質問を1〜2個に絞って聞く:

- 「このissueで解決したい問題・達成したい目標を教えてください。現状どんな困りごとがありますか？」
- 「機能追加の場合: 誰がどんなシーンで使いますか？」
- 「バグの場合: どんな操作をすると発生しますか？期待する動作と実際の動作を教えてください」

#### ラウンド2: 詳細の深堀り

ラウンド1の回答をもとに、不明点や矛盾点を深堀りする:

- 「〇〇とおっしゃいましたが、具体的にはどういう状態になれば完了といえますか？」
- 「このissueが完了したとき、ユーザーはどんな操作ができるようになりますか？」
- 「既存の〇〇機能との違い・関係性はありますか？」
- 「このissueを後回しにした場合、どんな影響がありますか？（優先度の確認）」

#### ラウンド3: リスク・依存関係の確認

必要に応じてさらに深堀りする:

- 「このissueを実装するにあたって、先に完了すべき別のissueはありますか？」
- 「データベース変更やAPI変更など、破壊的変更が含まれますか？」
- 「テストはどこまで書く必要があると思いますか？」

**インタビューの原則:**
- 一度に複数質問しない（最大2問まで）
- ユーザーの言葉を繰り返して確認する「〇〇ということですね？」
- ユーザーが気づいていない観点（受け入れ基準・リスク）を引き出す
- 「なぜ？」「具体的には？」「完了の定義は？」を意識する

### Step 3: Issueの草稿作成とレビュー

収集した情報をもとに `.claude/ISSUE_TEMPLATE.md` のフォーマットで草稿を作成し、AskUserQuestion でユーザーに確認する:

```
以下の内容でIssueを作成します。修正点があれば教えてください。

タイトル: [タイトル]

## Overview
[概要]
...
```

ユーザーの承認を得てから Step 4 に進む。

### Step 4: GitHub Issueの作成

`gh` コマンドでIssueを作成する:

```bash
gh issue create \
  --title "<タイトル>" \
  --body "<ISSUE_TEMPLATEフォーマットの本文>"
```

**本文のフォーマット** (`.claude/ISSUE_TEMPLATE.md` に準拠):

```markdown
## Overview
[1-2文で概要]

## Background
[背景・理由]

## Acceptance Criteria
- [ ] 基準1
- [ ] 基準2
- [ ] 基準3

## File Locations
- `src/[path]` - [説明]
- `tests/[path]` - [説明]

## Testing Requirements
- ユニットテスト: [必要なテスト]
- 統合テスト: [該当する場合]
- E2Eテスト: [該当する場合]

## Risk Assessment
- risk: [low / medium / high]
- 理由: [リスク判定の根拠]

## Dependencies
- 依存: [他のIssue]
- ブロック: [このIssueに依存するIssue]
```

作成完了後、IssueのURLをユーザーに伝える。

## 注意事項

- インタビューは日本語で行う
- Acceptance Criteriaは「具体的・テスト可能」な形式で3つ以上書く
- File Locationsは実際のプロジェクト構造を参照して記載する
- リスク判定の根拠を必ず明記する
